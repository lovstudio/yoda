# Yoda 商业计划书（BP）大纲

> 版本：v0.1（2026-07-21）
> 定位：面向独立开发者、技术创始人、小型 AI 原生团队，以及潜在合作方/投资人的商业计划书母稿。
> 写作原则：已验证事实与商业假设分开；所有市场、增长、收入数据均注明口径和日期；执行摘要最后写。

## 一句话定位

Yoda 是本地优先、供应商中立的 Agent Workspace 与 Harness 控制平面，让一个开发者能够像工程经理一样，编排多个编码 Agent 完成从任务、执行、隔离、审查到发布的完整交付闭环。

## 0. 使用方式与建议篇幅

- 正式商业计划书建议控制在 15–20 页，附录单列产品截图、数据口径和财务模型。
- 面向投资人时突出市场、增长与规模化路径；面向合作方时突出产品能力、生态接口与合作诉求。
- 本大纲中的“已验证”仅表示可由当前仓库材料支持，不代表市场或经营数据已经审计。
- 标记为“待验证”的内容在进入正式 BP 前必须补充来源、计算过程或负责人确认。

## 1. 执行摘要（最后撰写，正式稿 1 页）

严格压缩为五个要素：

1. **Yoda 做什么**：本地优先的多 Agent 编程工作台，统一管理多种编码客户端、任务、会话、Skills、Hooks、Memory、Worktree、Diff、Review 与 CI/CD。
2. **为谁解决什么问题**：服务同时使用两个及以上编码 Agent 的独立开发者、技术创始人和小型 AI 原生团队，解决工具碎片化、上下文不可见、并行任务难管理、工程交付仍依赖人工盯守的问题。
3. **解决方案与差异**：不绑定模型和厂商，以可观察、可恢复、可复用的 Harness 为核心，把多个 Agent 接入同一个交付工作区；同时支持主分支直跑和 Worktree 隔离两种工作流。
4. **市场机会**：用一条经过来源核验的 TAM/SAM/SOM 数字概括机会，避免直接套用宽泛的“生成式 AI 市场规模”。
5. **当前诉求**：根据实际用途填写融资额、渠道合作、生态伙伴、设计伙伴或未来 12 个月的内部经营目标。

### 本节待补证据

- 最新 GitHub 下载、Stars、Forks、独立 Cloner、活跃用户和留存数据，统一注明统计日期与是否去重。
- Relay 试用、激活、续费与付费转化数据。
- 若对外融资：融资金额、出让比例、资金用途和 18–24 个月里程碑。

## 2. 公司与项目概况

### 2.1 基本信息

- 项目名称：Yoda（Your Orchestra of Delegated Agents）。
- 产品形态：Electron 桌面应用 + Mobile/Gateway + Relay 云连接能力。
- 开源许可：Apache-2.0。
- 覆盖平台：macOS、Windows、Linux；移动端基于 Expo。
- 经营主体、成立时间、所在地、股权结构：**待创始人确认并与工商材料对齐**。
- 当前阶段：建议表述为“产品已发布、开源增长与商业化验证阶段”；是否属于早期收入阶段需用真实收入确认。

### 2.2 使命与愿景

- 使命：把编码 Agent 与真实工程世界之间的 Harness 做成可拥有、可观察、可改进的产品。
- 愿景：让一个开发者像工程经理一样，稳定指挥一支 Agent 团队持续交付。
- 组织策略：以一人公司/小团队为起点，通过 Agent、自动化、开源社区和生态伙伴放大交付能力。

## 3. 问题、解决方案与 Why Now

### 3.1 核心问题

围绕一个主问题展开，而不是罗列功能：模型能力快速提高，但真实软件交付仍被割裂的 Harness 与人工协调拖慢。

建议用三组可量化痛点支撑：

1. **工具与状态碎片化**：多个 Agent 分散在终端、IDE 或网页中，开发者需要频繁切换目录、窗口、会话和账号。
2. **上下文资产不可见**：Prompt、Skills、Hooks、Memory、模型配置和权限分散在不同客户端，难以观察、迁移、复用和调试。
3. **工程闭环依赖人工**：任务拆分、分支隔离、并行执行、Diff 审查、测试、CI、合并与归档之间缺少统一状态模型。

正式稿需补：

- 10–20 位目标用户访谈中的原话、频率和角色。
- 从任务启动到合并的时间分布，以及使用 Yoda 前后的对照数据。
- 多 Agent 并行时发生上下文重述、分支冲突、会话丢失和人工等待的频率。

### 3.2 Yoda 的解决方案

按用户旅程描述，而不是按功能菜单描述：

```text
Issue / Prompt
  → 选择 Agent、模型与运行模式
  → 主分支直跑或 Worktree 隔离
  → 多会话并行、对比、互审或团队协作
  → Diff、测试与 CI 状态验证
  → 合并、归档与归档前 Skill 自动收尾
  → 会话、Memory 与交付摘要可恢复复用
```

已可由仓库材料支持的能力：

- 统一编排 31 种编码客户端。
- normal、brainstorm、compare、review、team 五种运行模式。
- 自定义 Agent、Skills、Hooks、Memory 与会话上下文观测。
- Git Worktree 隔离、Diff/Review、工单接入和 CI/CD 状态联动。
- 本地 SQLite 存储，Yoda 默认不上传用户代码。
- Mobile/Gateway、Relay、SSH 远程项目与跨平台安装包。
- Yoda Build：从自然语言启动构建任务，并把生成的子 App 保存到 Library / Apps 继续迭代。

### 3.3 Before / After

| 环节 | 使用前 | 使用 Yoda 后 | 正式稿指标 |
| --- | --- | --- | --- |
| 多任务启动 | 手工开终端、切目录、复制上下文 | 从统一入口选择 Agent 与模式并行启动 | 启动时间、并发任务数 |
| 任务隔离 | 手工维护分支，容易互相污染 | 主分支直跑与 Worktree 隔离并存 | 冲突率、清理时间 |
| 上下文恢复 | 跨工具重新解释背景 | 会话、Skills、Memory 与交付摘要关联 | 恢复时间、重述次数 |
| 质量验证 | 人工轮询输出与测试 | Diff、Review、测试和 CI 状态进入同一闭环 | 缺陷率、审查时间 |
| 发布收尾 | Git、CI、文档与发布工具之间切换 | 归档前 Skill 和流程自动化 | 交付周期、遗漏率 |

### 3.4 Why Now

- 编码 Agent 从单一对话工具演进为可长时间运行、可并行的执行者，协调成本成为新瓶颈。
- Claude Code、Codex、Gemini 等客户端快速增加，开发者需要供应商中立的控制层，而不是再次绑定单一模型。
- Agent 的差异越来越取决于 Harness：上下文、工具、Skills、Hooks、Memory、权限和交付流程。
- 企业与专业开发者同时提高了对代码本地性、审计性、可恢复性和供应商可替换性的要求。

以上趋势必须在正式稿中补充近 12 个月的一手或权威来源。

## 4. 市场分析

### 4.1 目标客户

优先聚焦两个细分群体：

1. **独立开发者与技术创始人**：同时使用多个编码 Agent，需要用更少人力完成从需求到发布的完整交付。
2. **2–20 人 AI 原生研发团队**：需要统一 Agent 工作流、权限、审查、远程接入与可复用组织资产。

次级客户可包括 Agent 咨询/交付团队、开发者工具团队和对本地部署敏感的专业组织；正式 BP 不应同时把所有开发者都列为首要客户。

### 4.2 TAM / SAM / SOM 方法

- **TAM**：全球专业软件开发者中，为 AI 编码工具或开发者生产力工具付费的人群 × 年均工具预算。
- **SAM**：已使用两个及以上编码 Agent、重视本地工作流，并位于 Yoda 当前可服务平台和语言市场的人群 × 可实现年客单价。
- **SOM（3 年）**：按可触达渠道的流量、激活率、付费转化率和留存率做自下而上测算，不直接按 TAM 百分比拍脑袋。
- 每个数字均附来源、年份、地域、去重方式和计算公式；至少给出保守/基准/乐观三种情景。

### 4.3 市场趋势

- Agentic coding 从代码补全转向任务级、仓库级和团队级执行。
- 开发者开始组合多个模型与客户端，单一厂商锁定与资产迁移成为痛点。
- 本地优先、数据安全和可审计性成为专业用户的购买条件。
- Skills、Hooks、Memory 和 Agent 工作流开始形成可复用、可分发的生态资产。

## 5. 竞争格局与竞争楔子

### 5.1 建议比较的四类方案

- 单一厂商 Agent IDE / CLI。
- 云端自主编码 Agent。
- 终端复用、脚本和手工 Worktree 流程。
- 多 Agent 编排框架或工作台。

正式稿选取 3–5 个真实竞品，按下列维度建立矩阵：供应商中立、本地优先、多客户端并行、会话恢复、Worktree、Harness 可观测、Review/CI 闭环、移动/远程、团队治理、价格。

### 5.2 Yoda 的竞争楔子

- **供应商中立**：统一承载不同 Agent CLI，用户可按任务选择工具并保留资产。
- **Harness 可观测**：不只管理聊天窗口，而是暴露 Skills、Hooks、Memory 和上下文链路。
- **真实工程闭环**：把 Issue、会话、Worktree、Diff、Review、CI、合并和归档连接起来。
- **本地优先**：代码默认留在用户机器；云能力围绕连接、同步与协作增值。
- **兼容个人效率与团队治理**：先服务一人公司的高频工作流，再向小团队权限与协作扩展。

### 5.3 诚实披露的竞品优势

- 单一厂商产品可能拥有更深的模型集成、默认体验和资金优势。
- 云端 Agent 可能拥有更强的托管算力和无人值守体验。
- IDE 类产品拥有成熟编辑器生态和庞大分发渠道。
- Yoda 需要证明跨客户端编排带来的价值足以抵消配置和学习成本。

## 6. 商业模式与收入模型

### 6.1 建议商业路径

```text
开源 Yoda Desktop 建立信任与获客
  → Relay / 云同步等个人增值能力
  → 团队席位、权限、审计与共享 Harness 资产
  → Skills / 工作流 Marketplace 与托管执行能力
```

### 6.2 收入流（按验证顺序）

1. **Relay Pass / 个人订阅**：跨网络安全连接桌面 Agent，验证个人用户持续付费意愿。
2. **Pro 云能力**：云同步、跨设备体验、增强自动化和历史保留；范围与价格待验证。
3. **Team 席位**：团队权限、共享 Agent/Skills/Memory、审计、策略与协作工作流。
4. **Marketplace 抽成**：第三方 Skills、模板和工作流交易；必须在供需两侧形成规模后启动。
5. **企业支持或私有化服务**：作为后期收入补充，避免早期被重交付拖入项目制。

### 6.3 底层收入公式

- 月新增付费用户 = 有效触达 × 安装率 × 激活率 × 试用转付费率。
- 月经常性收入 = 各套餐活跃付费用户 × 对应月均客单价。
- LTV = 月均毛利 × 平均付费生命周期。
- CAC = 获客相关现金支出 ÷ 同期新增付费用户。
- 回本周期 = CAC ÷ 单用户月均毛利。

正式稿需明确：免费与付费边界、套餐价格、退款/续费规则、支付成本、Relay 基础设施成本和开源核心不商业化的边界。

## 7. 运营计划

### 7.1 产品交付

- Desktop 通过 GitHub Releases 和官网安装包分发。
- 版本更新、签名、公证、跨平台构建和镜像分发由 CI/CD 承担。
- 用户代码与本地状态默认留在本机；Relay、账号和云同步明确区分数据边界。
- 文档、Issue、更新日志和社区内容共同承担上手、支持与反馈收集。

### 7.2 一人公司关键流程

1. 用户反馈与问题归类。
2. Feature → Task → Agent 执行 → Review → Verification → Release。
3. 跨平台打包、更新源与公开资产验证。
4. Relay/云服务可用性、成本与安全监控。
5. 内容发布、案例沉淀和用户访谈。

### 7.3 自动化与外包边界

- 优先自动化测试、构建、发布、更新源验证、支持分类和经营数据看板。
- 当支持工时、平台兼容性或内容生产超过预设阈值时，引入兼职/外包伙伴。
- 产品方向、架构、安全边界和关键用户访谈继续由创始人直接负责。

## 8. 营销与销售计划

### 8.1 定位语

面向同时使用多个编码 Agent 的独立开发者与 AI 原生小团队，Yoda 是本地优先、供应商中立的 Agent Workspace；它把分散的客户端、Harness 资产和工程交付流程统一到一个可观察、可恢复的工作区。

### 8.2 优先渠道

1. **开源与开发者社区**：GitHub、Release、Issue、真实用户案例和可复现工作流。
2. **创始人内容**：围绕 Harness Engineering、一人公司、多 Agent 交付和 Build in Public 持续输出。
3. **生态合作**：Agent CLI、MaaS、模型平台、Skill 作者和开发者社区的联合内容与集成。

### 8.3 销售漏斗

```text
内容/社区触达
  → 官网或 GitHub
  → 下载与首次启动
  → 完成首个真实任务
  → 7 日内完成第二个任务或启用多 Agent/远程能力
  → Relay/Pro 试用
  → 付费与续费
```

正式稿为每一层补充当前基线、12 个月目标、责任渠道与实验计划。北极星指标建议从“每周完成并验证的 Agent 交付任务数”中验证，而不是只看下载或 Stars。

## 9. 财务预测

### 9.1 模型结构

- 第 1 年按月，第 2–3 年按季度。
- 至少包含保守和基准情景；乐观情景单列，不混入经营承诺。

## 10. 风险评估

正式稿按“概率 × 影响 × 缓解措施 × 预警指标”维护：

| 风险 | 概率 | 影响 | 缓解措施 | 预警指标 |
| --- | --- | --- | --- | --- |
| 单一创始人依赖与倦怠 | 中高 | 高 | 自动化发布与支持；关键流程文档化；建立兼职伙伴池 | 交付延期、支持积压、连续高工时 |
| Agent CLI/API 高频变化 | 高 | 高 | 可扩展 registry；兼容性测试；按使用量划分维护优先级 | 集成故障率、修复时长 |
| 大厂把编排能力内置 | 高 | 高 | 坚持供应商中立、本地优先和跨客户端资产层 | 多客户端用户占比下降 |
| 用户认可产品但不付费 | 中高 | 高 | 先验证 Relay/Pro 的刚需与续费，再扩展商业范围 | 试用转付费、续费率低 |
| 开源与付费边界失衡 | 中 | 高 | 明确核心开源、云连接/团队治理付费的原则 | 社区反弹、付费功能使用低 |
| 安全、隐私或供应链事件 | 中 | 高 | 最小权限、本地默认、签名公证、依赖审计、事件响应流程 | 异常访问、签名/更新失败 |
| 多平台与 31 客户端维护成本过高 | 高 | 中高 | 按活跃使用分级支持；自动化回归；社区共建 | 平台缺陷积压、兼容性 SLA 下降 |
| Marketplace 冷启动 | 中高 | 中 | 在交易前先做可分享目录、官方模板和作者激励 | 有效供给、安装与复用率低 |

## 11. 12–24 个月里程碑

### 阶段 A：证明核心使用价值（0–6 个月）

- 明确 ICP，并完成目标用户访谈与 cohort 追踪。
- 证明用户能持续通过 Yoda 完成真实任务，而不只是下载安装。
- 建立激活、周留存、任务完成和多 Agent 使用的基础数据口径。

### 阶段 B：证明个人付费（6–12 个月）

- 验证 Relay/Pro 的试用、转化、续费与毛利。
- 固化 2–3 个高频、可传播的付费场景。
- 将支持、发布和经营看板自动化，控制一人公司运营负担。

### 阶段 C：证明团队与生态扩张（12–24 个月）

- 与设计伙伴验证 Team 权限、审计和共享 Harness 资产。
- 建立 Skill/工作流的发布、信任、安全扫描与发现机制。
- 在供需数据成熟后决定是否启动 Marketplace 交易和抽成。

每个阶段应设置继续、调整或停止的量化门槛，避免仅以功能发布作为完成标准。

- 竞品功能、价格、许可与数据边界，记录检查日期。

- 公司主体、团队履历、收入与成本；经营数据与个人/其他业务严格分开。

## 13. 维护规则

- 前 6 个月每月更新一次，之后至少每季度更新一次。
- 所有未证实内容标记为“假设”，并记录验证方式、负责人和截止日期。
- 每次保留旧版本，更新日志说明哪些假设被证实、推翻或调整。
- 正式对外前请 2–3 位目标用户、创业者或顾问分别从产品价值、商业模型和财务口径审阅。
