# Yoda 商业计划书（BP）大纲

> 版本：v0.3（2026-07-21）
> 阅读对象：投资人，不预设读者具备 AI、编程或开发者工具背景。
> 写作原则：已验证事实与商业假设分开；所有市场、增长、收入数据均注明口径和日期；执行摘要最后写。

## 一句话定位

**人人都能指挥一支 Agent 团队，自由创造。**

下一层解释：AI Agent 是能够持续执行任务的 AI 工作成员。Yoda 把 Codex、Claude 等 Agent 组织成一支可以被人指挥的创作团队，让不具备专业编程能力的人，也能把想法变成 App、数字产品以及文字、图像、视频等 AIGC 内容。Yoda 当前从代码与 App 创作切入，长期服务所有用 AI 创造的人。

## 0. 使用方式与建议篇幅

- 正式商业计划书建议控制在 15–20 页，附录单列产品截图、数据口径和财务模型。
- 只面向投资人写作，优先回答“用户为什么需要、市场有多大、凭什么是 Yoda、如何增长和赚钱”。
- 第一次出现 Agent、Harness、MaaS 等术语时，必须用普通商业语言解释，不假设投资人懂 AI。
- 本大纲中的“已验证”仅表示可由当前仓库材料支持，不代表市场或经营数据已经审计。
- 标记为“待验证”的内容在进入正式 BP 前必须补充来源、计算过程或负责人确认。

## 1. 执行摘要（最后撰写，正式稿 1 页）

严格压缩为五个要素：

1. **Yoda 做什么**：Yoda 是面向 AI 时代创造者的 Agent 创作平台。它把不同 AI Agent 组织成一支可被指挥的团队，并通过一套统一工作系统管理它们的上下文、能力、协作、审查与交付，让人能够持续把想法变成作品和产品。
2. **为谁解决什么问题**：长期服务所有希望借助 AI 创造的人；早期聚焦独立创作者、个人创业者、技术创始人和小型 AI 原生工作室。用户不需要先成为专业开发者，但仍需要清楚表达目标、观察过程、作出判断并验收成果。
3. **解决方案与差异**：Yoda 不把 AI 当作一个聊天助手，而是把 Codex、Claude 等 Agent 组织成可分工、可协作、可干预的创作团队；当前从 App 与代码创作切入，逐步扩展到数字产品和 AIGC 工作流。
4. **市场机会**：Visa 2025 报告估计全球约有 2.07 亿创作者；按每年 120 美元的基础工具支出测算，理论 TAM 为 248 亿美元，当前可服务市场约 24.8 亿美元，三年可获取市场目标约 240 万美元 ARR；其中 TAM/SAM 包含待验证假设，计算口径见市场分析。
5. **当前诉求**：融资 200 万元人民币或 30 万美元，出让 10% 股权；资金主要用于技术研发和市场推广，支撑未来 18–24 个月产品完善、增长验证与商业化。

- 产品形态：Electron 桌面应用 + Mobile/Gateway + Relay 云连接能力。

### 2.2 使命与愿景

- 使命：降低创造的技术门槛，让每个人都能把想法变成真实作品和产品。
- 愿景：人人都能指挥一支 Agent 团队，自由创造。
- 组织策略：以一人公司/小团队为起点，通过 Agent、自动化、开源社区和生态伙伴放大交付能力。

## 3. 问题、解决方案与 Why Now

### 3.1 核心问题

围绕一个主问题展开，而不是罗列功能：大模型已经显著降低了生成代码和内容的门槛，但把一个想法持续变成可用、可迭代、可发布的作品，仍然要求人理解复杂工具并亲自协调多个 AI 工作过程。

建议用三组可量化痛点支撑：

1. **创造仍受技术门槛限制**：普通人可以用自然语言提出想法，却仍要学习终端、IDE、代码仓库、部署和多个 AI 工具，才能真正把结果做出来。
2. **Agent 能力与状态碎片化**：Prompt、Skills、Hooks、Memory、模型配置和权限散落在不同客户端与文件中，创作者难以理解、组合、复用和迁移。
3. **人工盯守体验不友好**：创造过程在可预见的未来仍需要人提供品味、判断、干预和验收。问题不是“还要人参与”，而是人必须在多个窗口低效轮询，难以快速理解进度和风险。

正式稿需补：

- 10–20 位不同类型创作者访谈中的原话、频率和角色，至少覆盖非技术创作者、个人创业者与技术创作者。
- 从任务启动到合并的时间分布，以及使用 Yoda 前后的对照数据。
- 使用单 Agent 与多 Agent 时发生上下文重述、会话丢失、反复切换窗口和人工等待的频率。

### 3.2 Yoda 的解决方案

先展示多种创作与开发范式，再描述 Yoda 如何用统一 Harness 承接它们。分支或 Worktree 只是可选的隔离开关，不作为核心卖点：

```text
一个想法 / 一个创作目标
  → 选择单 Agent、Vibe Coding、多智能体协作或 Build 模式
  → 按需要选择模型、MaaS 平台与隔离方式
  → Yoda 统一管理 Session、Skills、Hooks、Memory 与执行状态
  → 创作者随时观察、干预、比较和 Review
  → Diff、测试与 CI/CD 验证
  → 继续迭代、发布或沉淀为可复用的 App、产品与工作流
```

已可由仓库材料支持的能力：

- 统一接入 31 种 Agent Client；既可只使用一个 Agent，也可按任务组合多个 Agent。
- 覆盖普通开发、Vibe Coding、brainstorm、compare、review、team 和 Build 等多种开发范式。
- 自定义 Agent、Skills、Hooks、Memory 与会话上下文观测。
- Git Worktree 隔离、Diff/Review、工单接入和 CI/CD 状态联动；是否启用隔离由用户按任务决定。
- 本地与云端混合架构：项目与代码默认留在用户设备，模型、MaaS、账号、云同步和 Relay 按需使用云服务。这里的“本地”指数据与执行边界，不代表大模型在本机运行。
- Mobile/Gateway、Relay、SSH 远程项目与跨平台安装包，支持跨设备继续观察和干预 Agent 工作。
- Yoda Build：从自然语言启动构建任务，并把生成的子 App 保存到 Library / Apps 继续迭代。

### 3.3 Before / After

| 环节 | 使用前 | 使用 Yoda 后 | 正式稿指标 |
| --- | --- | --- | --- |
| 开始创造 | 先学习和拼接不同技术工具 | 从一个想法直接选择适合的 Agent 创作方式 | 首次价值时间、创作启动时间 |
| 过程盯守 | 在多个窗口反复轮询、难以判断下一步 | 在一个指挥台观察状态并随时干预 | 切换次数、人工等待时间 |
| 上下文恢复 | 跨工具重新解释背景 | 会话、Skills、Memory 与交付摘要关联 | 恢复时间、重述次数 |
| 质量验证 | 人工轮询输出与测试 | Diff、Review、测试和 CI 状态进入同一闭环 | 缺陷率、审查时间 |
| 创作与迭代 | AI 生成一次结果后，仍需手工拼接后续流程 | 从想法、生成、审查到发布和继续迭代保持连续 | 交付周期、二次迭代率 |

### 3.4 Why Now

- Codex 等 Agent 从代码补全演进为可承担完整任务的执行者，人的角色正从“亲自完成每一步”转向“提出目标、提供上下文、判断和验收”。
- GPT、Claude、Codex 等模型和 Agent 同时降低了代码、产品与内容创作门槛，创作能力开始从专业团队扩散到每一个有想法的人。
- Agent 的差异越来越取决于 Harness：上下文、工具、Skills、Hooks、Memory、权限和交付流程。
- [Adobe 2026 Creators’ Toolkit Report](https://news.adobe.com/news/2026/06/creators-toolkit-report-2026)显示，75% 的受访创作者认为创意 AI 已融入或成为其工作流的必要部分，但 85% 仍认为最终创意决策应由人完成。这正是 Yoda“Agent 团队执行、人负责指挥与品味”的市场基础。

以上趋势必须在正式稿中补充近 12 个月的一手或权威来源。

## 4. 市场分析

### 4.1 目标客户

用“终局市场—早期用户—产品滩头阵地”三层定义，避免把当前用户等同于长期市场：

1. **终局市场——所有 AI 时代的创造者**：任何希望把想法变成 App、数字产品、互动体验或 AIGC 内容的人，不以是否具备专业编程能力划分。
2. **早期核心用户——AI 原生创作者**：独立创作者、个人创业者、技术创始人和 2–20 人创作工作室；他们主动组合 AI 工具，并对创造效率、可控性和成果质量有持续需求。
3. **产品滩头阵地——代码与 App 创作**：Yoda 当前 Harness、Diff Review、CI/CD 与 Agent Client 集成在这一场景最成熟，因此先在这里证明留存与付费，再逐步降低技术门槛、扩展创作类型。

这里的“创造者”不是“开发者”的营销替换词。正式 BP 必须用非技术用户能否独立完成首次作品、创作类型占比和跨人群留存，证明市场扩展真实发生。

### 4.2 TAM / SAM / SOM 基准测算

| 市场 | 基准数字 | 计算方式 | 口径说明 |
| --- | ---: | --- | --- |
| TAM | 248 亿美元/年 | 2.07 亿全球创作者 × 120 美元年费 | 创作者人数来自 Visa 2025 报告；假设每位创作者都可能购买一套基础 AI 创作工具，因此是理论上限 |
| SAM | 24.8 亿美元/年 | 2.07 亿 × 10% AI 原生创作者适配率 × 120 美元年费 | 10% 是对近期会主动使用 Agent 创作产品或内容人群的待验证假设 |
| SOM（3 年） | 240 万美元 ARR | 2 万个付费账户 × 120 美元年费 | 是经营目标，不是当前收入或市场份额事实 |

- [Visa 2025 Creator Report](https://corporate.visa.com/en/solutions/commercial-solutions/knowledge-hub/2025-visa-creators-report.html)估计全球约有 2.07 亿创作者，并预计创作者经济到 2027 年达到约 5,000 亿美元。该规模只用于说明宏观背景，不直接当作 Yoda 的软件收入 TAM。
- [SlashData](https://www.slashdata.co/research/developer-population)估计全球有 4,840 万活跃开发者。这部分人群不再定义 Yoda 的 TAM，而是 Yoda 当前产品最容易触达和验证价值的首个滩头市场。
- 120 美元年费与当前 Relay 990 credits/30 天（约 9.9 美元/月）的价格锚点接近，正式模型仍需区分免费版、Creator Pro、Relay 和 Studio。
- TAM 假设所有创作者都可能购买基础 AI 创作工具，SAM 又假设其中 10% 属于近期适配人群；两者都必须通过用户研究、定价测试和真实转化继续缩小误差，不能直接当收入机会。

### 4.3 市场趋势

- **创造民主化**：自然语言逐步成为代码、产品和内容的新创作界面，非技术人群首次能够直接参与软件创造。
- **从助手到团队**：Agent 从回答问题升级为承担任务，人需要的产品从单个聊天框转向任务分工、过程观察和结果验收。
- **创作者即企业**：创作者从内容发布者扩展为独立产品经营者，需要覆盖生产、分发、变现和跨设备协作的基础设施。
- **可复用的创作资产**：Skills、Hooks、Memory 和 Agent 工作流开始成为可组合、可分发的数字生产资料。

## 5. 竞争格局与竞争楔子

### 5.1 建议比较的四类方案

- 单一厂商 Agent IDE / CLI。
- 面向非技术用户的 AI App Builder 与无代码平台。
- GPT、Claude 等通用 AI 助手和 AIGC 创作工具。
- 云端自主编码 Agent。
- 多 Agent 编排框架或工作台。

正式稿选取 3–5 个真实竞品，按下列维度建立矩阵：创作者门槛、输出类型、供应商中立、单/多 Agent、创作范式、会话恢复、Harness 可观测、结果验证、移动/远程、团队协作、价格。

### 5.2 Yoda 的竞争楔子

- **供应商中立**：统一承载不同 Agent CLI，用户可按任务选择工具并保留资产。
- **Harness 可观测**：不只管理聊天窗口，而是暴露 Skills、Hooks、Memory 和上下文链路。
- **从想法到真实作品**：当前先把 App 创作中的会话、Diff、Review、CI、发布和继续迭代连接起来，再扩展到更多数字创作流程。
- **本地与云端混合架构**：项目和代码默认留在用户设备，模型、MaaS、同步与 Relay 等云能力按需接入，边界清楚且不误导为“本地大模型”。
- **人的品味始终在环**：Yoda 不追求让 AI 完全替代创作者，而是让人持续掌握目标、选择、干预和最终决定。

### 5.3 诚实披露的竞品优势

- 单一厂商产品可能拥有更深的模型集成、默认体验和资金优势。
- 云端 Agent 可能拥有更强的托管算力和无人值守体验。
- IDE 类产品拥有成熟编辑器生态和庞大分发渠道。
- Yoda 需要证明跨客户端编排带来的价值足以抵消配置和学习成本。

## 6. 商业模式与收入模型

### 6.1 建议商业路径

```text
开源 Yoda Desktop 建立信任与获客
  → Relay / 云同步等个人增值能力
  → Studio 席位、权限、审计与共享 Harness 资产
  → Skills / 工作流 Marketplace 与托管执行能力
```

### 6.2 收入流（按验证顺序）

1. **Relay Pass / 创作者订阅**：跨网络安全连接桌面 Agent，验证个人创作者持续付费意愿。
2. **Creator Pro 云能力**：云同步、跨设备创作、增强自动化、作品历史和发布能力；范围与价格待验证。
3. **Studio 席位**：工作室权限、共享 Agent/Skills/Memory、审计、品牌资产与协作工作流。
4. **Marketplace 抽成**：第三方 Skills、模板和工作流交易；必须在供需两侧形成规模后启动。
5. **企业支持或私有化服务**：作为后期收入补充，避免早期被重交付拖入项目制。

### 6.3 底层收入公式

- 月新增付费用户 = 有效触达 × 安装率 × 激活率 × 试用转付费率。
- 月经常性收入 = 各套餐活跃付费用户 × 对应月均客单价。
- LTV = 月均毛利 × 平均付费生命周期。
- CAC = 获客相关现金支出 ÷ 同期新增付费用户。
- 回本周期 = CAC ÷ 单用户月均毛利。

正式稿需明确：免费与付费边界、套餐价格、退款/续费规则、支付成本、Relay 基础设施成本和开源核心不商业化的边界。

## 7. 运营计划

### 7.1 产品交付

- Desktop 通过 GitHub Releases 和官网安装包分发。
- 版本更新、签名、公证、跨平台构建和镜像分发由 CI/CD 承担。
- 用户代码与本地状态默认留在本机；Relay、账号和云同步明确区分数据边界。
- 文档、Issue、更新日志和社区内容共同承担上手、支持与反馈收集。

### 7.2 一人公司关键流程

1. 用户反馈与问题归类。
2. Feature → Task → Agent 执行 → Review → Verification → Release。
3. 跨平台打包、更新源与公开资产验证。
4. Relay/云服务可用性、成本与安全监控。
5. 内容发布、案例沉淀和用户访谈。

### 7.3 自动化与外包边界

- 优先自动化测试、构建、发布、更新源验证、支持分类和经营数据看板。
- 当支持工时、平台兼容性或内容生产超过预设阈值时，引入兼职/外包伙伴。
- 产品方向、架构、安全边界和关键用户访谈继续由创始人直接负责。

## 8. 营销与销售计划

### 8.1 定位语

**人人都能指挥一支 Agent 团队，自由创造。** Yoda 把分散的 AI 能力组织成一支可以分工、协作、被人持续指挥的创作团队，让每个有想法的人都能完成过去需要专业技术团队才能完成的作品。

### 8.2 优先渠道

1. **AI 创作者社区**：用“一个人指挥 Agent 团队完成作品”的真实案例触达独立创作者、个人创业者和小型工作室。
2. **开源与技术创作者社区**：通过 GitHub、Release、Issue 和可复现工作流守住当前最强的产品口碑与早期用户来源。
3. **创始人内容与生态合作**：围绕一人公司、Agent 创作和 Build in Public 持续输出，并与模型平台、MaaS、Skill 作者和创作者平台联合推广。

### 8.3 销售漏斗

```text
内容/社区触达
  → 官网或 GitHub
  → 下载与首次启动
  → 完成并保存首个真实作品
  → 7 日内继续迭代或完成第二个作品
  → Relay/Pro 试用
  → 付费与续费
```

正式稿为每一层补充当前基线、12 个月目标、责任渠道与实验计划。北极星指标建议验证“每周由 Agent 团队完成并被创作者验收的作品数”，而不是只看下载、Stars 或 Agent 调用次数。

## 9. 财务预测

### 9.1 模型结构

- 第 1 年按月，第 2–3 年按季度。
- 至少包含保守和基准情景；乐观情景单列，不混入经营承诺。
- 收入：Relay/Creator Pro、Studio、Marketplace、企业支持分别建模。
- 成本：支付手续费、Relay/云资源、监控、代码签名/发布、SaaS 工具、市场投放、外包和人员。
- 输出：收入、直接成本、毛利、运营费用、净利润/亏损和期末现金。

### 9.2 必算阈值

- 月度盈亏平衡所需付费用户数。
- Relay/云能力的单用户边际成本与毛利率。
- 创始人投入和现金储备对应的可持续月数。
- 在付费转化、续费率或基础设施成本恶化时的现金流敏感性。

## 10. 风险评估

正式稿按“概率 × 影响 × 缓解措施 × 预警指标”维护：

| 风险 | 概率 | 影响 | 缓解措施 | 预警指标 |
| --- | --- | --- | --- | --- |
| 单一创始人依赖与倦怠 | 中高 | 高 | 自动化发布与支持；关键流程文档化；建立兼职伙伴池 | 交付延期、支持积压、连续高工时 |
| Agent CLI/API 高频变化 | 高 | 高 | 可扩展 registry；兼容性测试；按使用量划分维护优先级 | 集成故障率、修复时长 |
| 大厂把编排能力内置 | 高 | 高 | 坚持供应商中立、清晰数据边界和跨客户端 Harness 资产层 | 跨客户端用户占比下降 |
| 用户认可产品但不付费 | 中高 | 高 | 先验证 Relay/Pro 的刚需与续费，再扩展商业范围 | 试用转付费、续费率低 |
| 开源与付费边界失衡 | 中 | 高 | 明确核心开源、云连接/团队治理付费的原则 | 社区反弹、付费功能使用低 |
| 安全、隐私或供应链事件 | 中 | 高 | 最小权限、本地默认、签名公证、依赖审计、事件响应流程 | 异常访问、签名/更新失败 |
| 多平台与 31 客户端维护成本过高 | 高 | 中高 | 按活跃使用分级支持；自动化回归；社区共建 | 平台缺陷积压、兼容性 SLA 下降 |
| Marketplace 冷启动 | 中高 | 中 | 在交易前先做可分享目录、官方模板和作者激励 | 有效供给、安装与复用率低 |

## 11. 18–24 个月季度里程碑

以下是投资人版本的建议目标，用户数和收入目标需与完整财务模型再次校准：

| 时间 | 核心目标 | 关键交付与验证 |
| --- | --- | --- |
| 0–3 个月 | 产品功能完善、体验顺滑 | 打磨单 Agent、Vibe Coding、多智能体与 Build 主路径；降低非技术用户首次创作门槛；建立激活、作品完成、DAU、D7/D30 留存埋点；完成首轮 20 位跨类型创作者访谈 |
| 4–6 个月 | 建立可重复的自然增长 | 完善官网、创作者模板和真实作品案例；开始持续内容推广与社区合作；验证“首次完成真实作品”激活漏斗 |
| 7–9 个月 | 开始小规模投放与付费验证 | 基于前两季度数据逐步投放；迭代 Relay/Pro 定价；验证试用转付费、续费和单用户云成本 |
| 10–12 个月 | 找到 2–3 个高留存场景 | 聚焦可传播的核心场景；建立 cohort 看板和自动化运营；让增长、激活与付费形成可复现漏斗 |
| 13–15 个月 | 验证小型工作室需求 | 与设计伙伴测试共享 Agent、Skills、Memory、品牌资产、权限和审计；形成 Studio 版本最小闭环 |
| 16–18 个月 | 扩大市场推广 | 加大有效渠道投放和生态合作；扩展英文市场；验证创作者订阅与 Studio 席位的组合收入 |
| 19–21 个月 | 建立 Harness 生态 | 完善 Skill/工作流的发布、信任、安全扫描和发现；先验证供给、安装与复用，再决定交易抽成 |
| 22–24 个月 | 证明规模化能力 | 形成能够冲击 2 万付费账户、240 万美元 ARR 三年 SOM 的增长模型；验证增长效率、毛利和组织可持续性是否支持下一轮融资 |

每季度设置继续、调整或停止的量化门槛。前三个月不急于大规模投放，先把产品体验和数据基础做扎实；之后再根据真实留存逐步增加推广预算。
